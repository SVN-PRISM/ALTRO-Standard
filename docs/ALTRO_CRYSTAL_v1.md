# ALTRO Crystal — бинарный артефакт ядра

**Версия спецификации:** 1.0 (Zero-Copy / Cold-Start Optimized)  
**Имя файла:** `altro_crystal.bin`  
**Статус:** неизменяемый (immutable) артефакт; обновление только заменой файла и bump версии.

---

## 1. Общее описание

`altro_crystal.bin` — неизменяемый бинарный файл семантической карты весов. Предназначен для мгновенного атомарного взвешивания токенов без использования тяжёлых LLM.

- Загрузка: один `ArrayBuffer` / отображение в память; доступ к матрицам через `Float32Array(buffer, byteOffset, length)` без парсинга protobuf/JSON на hot-path.
- Назначение: lookup токена → индекс строки → скалярные произведения со **13** эталонными центроидами доменов (семантические «координаты»).
- Целевой размер типичной сборки: порядка **15–20 MB** при `N ≈ 10k–15k`, `dim` 256–300 (зависит от выбранного D и словаря).

---

## 2. Структура (Layout)

Порядок блоков **с начала файла** (offsets в HEADER отсчитываются от байта 0 файла).

```text
┌─────────────────────────────────────────┐
│ HEADER (256 B, фиксированный)            │
├─────────────────────────────────────────┤
│ HASH_TABLE (open addressing)            │
├─────────────────────────────────────────┤
│ VECTORS — Float32 [N × dim] row-major   │
├─────────────────────────────────────────┤
│ CENTROIDS — Float32 [13 × dim]          │
├─────────────────────────────────────────┤
│ (опционально v1+) зарезервировано       │
└─────────────────────────────────────────┘
```

### 2.1. HEADER (256 байт)

| Смещение | Размер | Поле | Описание |
|----------|--------|------|----------|
| 0 | 4 | `magic` | `0x414C5452` (ASCII «ALTR»), little-endian |
| 4 | 2 | `spec_version` | Версия формата файла (= 1 для v1) |
| 6 | 2 | `flags` | Битовая маска (см. ниже) |
| 8 | 4 | `dim` | Размерность D (например 256 или 300) |
| 12 | 4 | `N_tokens` | Число строк словаря N |
| 16 | 8 | `hash_seed` | Seed для xxHash64 ключей (воспроизводимость) |
| 24 | 8 | `off_hash_table` | Байтовое смещение начала HASH_TABLE |
| 32 | 4 | `hash_slots` | Число слотов таблицы (≥ N, обычно ~1.3–1.5× N) |
| 36 | 4 | `slot_stride` | Размер одного слота в байтах (= 16: hash64 + index32 + padding) |
| 40 | 8 | `off_vectors` | Смещение начала матрицы VECTORS |
| 48 | 8 | `off_centroids` | Смещение начала матрицы CENTROIDS |
| 56 | 8 | `off_reserved` | 0 или смещение будущих блоков (UNK, invNorm, string pool) |
| 64 | 8 | `file_size` | Ожидаемый размер файла (опционально, 0 = не задано) |
| 72 | 8 | `checksum` | Контрольная сумма payload (xxHash64 или 0) |
| 80 | … | `reserved` | Padding до **256** байт (выравнивание, будущие поля) |

**Endianness:** все много-байтовые целые и float — **little-endian**.

**`flags` (рекомендуемые биты v1):**

- bit0: векторы строк **уже L2-нормированы** (dot с нормированными центроидами = косинус).
- bit1: зарезервирован.
- bit2: присутствует расширенный блок (через `off_reserved`).

**Выравнивание:** начало блоков `HASH_TABLE`, `VECTORS`, `CENTROIDS` — кратно **64 байтам** (удобство для SIMD / WASM).

### 2.2. HASH_TABLE — Open Addressing (xxHash64 → Index)

- Ключ lookup: **нормализованный токен** как UTF-8 байты (правила токенизации ALTRO задаются отдельно).
- Хеш: **xxHash64**(bytes, `hash_seed`) → `uint64`.
- Слот: минимум **16 байт**: `uint64 hash`, `uint32 row_index` (0 … N−1), `uint32 reserved` или `uint32 probe_tag`.
- Коллизии: **linear probing** (или иная схема, зафиксированная билдером) в пределах `hash_slots`.
- Пустой слот: `hash == 0` и `row_index == 0xFFFFFFFF` (или иная договорённость билда — главное, чтобы загрузчик и билдер совпадали).
- Успех: найден слот с равным хешем **и** подтверждённым ключом, если билдер хранит вторичную проверку; в минимальной схеме v1 допускается только hash+index при отсутствии коллизий хеша на датасете (билд обязан гарантировать уникальность hash для всех N ключей **или** добавить string pool в расширении).

Рекомендация для production: билд гарантирует уникальность xxHash64 по всем токенам **или** вводит блок верификации ключей в `off_reserved`.

### 2.3. VECTORS

- Тип: **Float32**, порядок **row-major**: строка `i` занимает байты `[off_vectors + i * dim * 4, …)`.
- Семантика: статическое семантическое пространство (например усечённые предобученные эмбеддинги).
- При `flags.bit0 = 1` строки **нормированы**; иначе ядро должно применять нормализацию или хранить `invNorm` в расширении (не часть минимального v1).

### 2.4. CENTROIDS

- Тип: **Float32**, матрица **[13 × dim]** в том же порядке, что и строки: домен `k` — смещение `off_centroids + k * dim * 4`.
- Центроиды **L2-нормированы** офлайн.
- Смысл 13 якорей соответствует доменной модели ALTRO (идентификаторы доменов — контракт продукта, не формата файла).

## 3. Механика сита

1. **Токенизация** входа по правилам ядра → нормализованный токен (UTF-8).
2. **Lookup** в HASH_TABLE: xxHash64 → пробинг → `row_index`.
3. **Dot product** строки `VECTORS[row_index]` с каждой из **13** строк `CENTROIDS` (при нормированных векторах — косинусная близость).
4. **Сравнение с порогом** политики Директивы / SovereignController (per-domain thresholds, top-k, маски) — вне формата файла, но обязательный этап конвейера.

**OOV (нет в таблице):** политика ядра по умолчанию — нулевой вклад или флаг OOV; опционально расширения: средний вектор, hash-embeddings (отдельный блок или sidecar — не обязательны в минимальном v1).

---

## 4. Гибридность

Поддержка внешних ONNX-моделей и аппаратных NPU через **стандартизированный интерфейс на базе `Float32Array`**:

- Кристалл отдаёт или принимает представление вектора длины `dim` как **подвид** одного непрерывного буфера (zero-copy view).
- Тяжёлый энкодер (ONNX / NPU) может вычислить вектор фразы/предложения в тот же `dim` (после выравнивания пространства — отдельный контракт версии модели), после чего те же **13** dot-продуктов с `CENTROIDS` выполняются тем же кодом, что и для статических строк кристалла.
- Так ядро остаётся единым для «лёгкого пути» (кристалл) и «тяжёлого пути» (внешний энкодер) без дублирования логики порогов.

---

## 5. Версионирование и доставка

- Файл **не патчится** на месте: выпуск новой семантики = новый `altro_crystal.bin` (и при необходимости новое имя или каталог версий).
- Загрузчик проверяет `magic`, `spec_version`, `dim`, `N_tokens`, опционально `checksum` и `file_size`.
- Центроиды при частых изменениях доменов допускается выносить в отдельный малый артефакт той же размерности; минимальный v1 допускает их хранение **внутри** этого файла после `VECTORS`.
